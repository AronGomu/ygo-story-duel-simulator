02 · MVP boundary

One complete duel product, not the story game

The release proves the hardest reusable foundation: a custom browser client can legally drive the current Project Ignis engine from initialization through a structured result.

Inside the cut line

  • One human player versus one deterministic, deck-specific opponent.
  • Bundled preset decks and a direct-to-duel launch flow.
  • Dedicated Worker owning synchronous ocgcore WASM.
  • Current standard-format card data, official scripts and system strings.
  • Typed prompts, legal responses, projected state and hidden-information controls.
  • Svelte controls and overlays plus a Phaser duel field.
  • Card image preload, lazy cache, card backs and deterministic fallback.
  • Restart, surrender, final result and downloadable diagnostics.
  • Automated asset, unit, integration and browser verification.

Explicitly deferred

  • Story chapters, dialogue, maps, relationships and progression.
  • Deck editor, collection, packs, rewards and shops.
  • General simulator menus, repositories and replay browser.
  • Generic AI or support for arbitrary user decks.
  • Online play, server authority, lobbies, chat and side decking.
  • Mobile-first polish and pixel-level EDOPro parity.
  • Bulk shipment of every high-resolution card image.

03 · Decisions to lock

Resolve these before the critical path expands

D1
Compatibility promise

Keep the accepted broad target: fixture and classify every current core prompt family, while only requiring end-to-end scenarios for the final preset decks. Reducing this to preset-reachable messages would shorten the MVP substantially, but changes the accepted product claim.

Scope driver
D2
Exact preset decks

Choose simple, stable 40-card lists that prove summon, set, activation, targeting, battle, damage and at least one chain. Avoid unusual zones and free-form announcements until the foundation is green.

Required
D3
Atomic upstream revision set

Pin compatible ocgcore-wasm, embedded ygopro-core, BabelCDB, CardScripts, Distribution strings and image metadata. Upgrades move as one tested snapshot.

Required
D4
Image provider and deployment posture

Use the existing YGOPRODeck URL convention for development unless availability or CORS testing fails. Public deployment remains blocked until redistribution, caching and IP posture are explicitly approved.

Release gate
D5
Primary browser and support floor

Use current Chromium for daily development. Firefox and Safari are release verification targets. Avoid JSPI, SharedArrayBuffer and cross-origin-isolation dependencies in the MVP.

Recommended
?

Largest correction opportunity: the existing MVP combines a focused preset-duel product with broad current-protocol conformance. That is defensible, but it is the main reason this is an 18–30 week effort rather than a small prototype.

04 · Macro roadmap

Seven gates from source move to release candidate

Each phase ends in observable proof. Work does not advance because files exist; it advances when the engine or browser demonstrates the required behavior.

Move→Freeze→Headless duel→Protocol and state→Controls and AI→UI→Hardening→Release
00.5–1 wk

Move and establish the repository

Small

Transfer the planning project into the implementation folder without losing history or the known-good asset baseline.

Outputs

  • Project files and Git metadata in the current folder.
  • Generated and cache directories remain ignored.
  • Install, tests, typecheck and asset verification run from the new location.
  • README-level developer commands reflect the final path.

Gate 0

Clean checkout reproduces the package install and all current verification commands. The generated snapshot either verifies after transfer or can be regenerated exactly.

11–2 wks

Freeze the playable slice

Medium

Turn architecture choices into immutable inputs and executable contracts before UI work begins.

Outputs

  • Vite, strict TypeScript and Svelte skeleton with Phaser installed but not yet driving rules.
  • Final player and opponent deck lists plus parsed .ydk fixtures.
  • Atomic revision manifest including the WASM core and all data sources.
  • Typed Worker command/event contract, result/error schema and run-trace schema.
  • Current message/prompt/response inventory generated from pinned sources.

Gate 1

Production build opens, a Worker boots, reports the pinned core version and rejects a mismatched snapshot before any duel starts.

22–4 wks

Prove the headless engine loop

Large · risk burn-down

Demonstrate the central technical claim before investing in the complete client: browser Worker plus synchronous callbacks can drive a deterministic legal duel.

Outputs

  • Owned or forked ocgcore-wasm adapter loaded only inside the Worker.
  • Active-duel card, script and string shards expanded into synchronous maps.
  • Duel creation, deck population, processing loop, responses and disposal.
  • Simple deterministic auto-responder for both sides.
  • Ordered trace with seed, revisions, messages, responses and fatal diagnostics.

Gate 2 · Go or re-plan

A fixed seed reaches MSG_WIN twice with the same ordered trace and no runtime Node API or asynchronous core callback. If this fails, stop UI development and re-plan the adapter.

35–8 wks

Build the protocol and state foundation

Extra large · critical path

Translate core binary semantics into stable domain prompts, events and immutable public state without leaking hidden information.

Outputs

  • Raw message parser and response encoders contained inside the Worker.
  • Prompt families classified, typed and backed by fixtures.
  • Opaque prompt/choice IDs with stale, duplicate and invalid-response rejection.
  • State projector for LP, turn, phase, zones, positions, chain, counts and result.
  • Query-based reconciliation and invariants against card duplication/loss.
  • Public-state projection that strips opponent secrets before postMessage.

Gate 3

The compatibility matrix has no unclassified pinned-core prompt type. Recorded fixtures produce expected prompts and state. Hidden opponent data never reaches the main thread.

42–4 wks

Add human control and the opponent policy

Large

Replace automatic responses with one transparent human decision path and one finite, reproducible policy for the bundled opponent deck.

Outputs

  • Debug prompt console or minimal Svelte action controls for every typed prompt.
  • Deck-specific priorities for phases, summons, activations, chains, targets and battle.
  • Mandatory fallback behavior and decision reason logging.
  • Hot-seat diagnostic mode using the same human prompt path.
  • Unit tests for opponent choices and stale-response safety.

Gate 4

A human can complete a duel without raw protocol knowledge. The opponent takes meaningful turns and answers every prompt reachable by its deck under fixed scenarios.

53–5 wks

Deliver the duel experience

Large

Build the usable, image-backed duel surface while keeping presentation strictly downstream of authoritative Worker state.

Outputs

  • Svelte loading, error, prompt, inspector, log, surrender and result surfaces.
  • Phaser board, zones, field cards, hand, selection states and essential movement feedback.
  • Small presentation-store boundary between Svelte and Phaser.
  • Active-deck image preload, cache keys, card backs and missing-image placeholders.
  • Accessible keyboard path and clear focus for prompt selection.
  • Responsive desktop layout for Chromium, Firefox and Safari targets.

Gate 5

The complete human flow works without debug controls. Correct public images appear in every required zone and prompt. Animation cannot block, duplicate or reorder responses.

63–5 wks

Close compatibility and reliability gaps

Extra large

Turn the successful vertical slice into a repeatable MVP that fails with evidence rather than hanging silently.

Outputs

  • Final preset decks and deterministic scenarios for their key interactions.
  • Both win directions, deck-out where applicable, surrender, restart and clean disposal.
  • Protocol conformance corpus for the broad accepted compatibility promise.
  • Downloadable diagnostics for unsupported messages, missing assets and engine faults.
  • Snapshot cache validation, atomic activation and rollback to known-good data.
  • Leak checks for repeated restarts and Worker termination.

Gate 6

Multiple full duels complete to core-provided results. All supported prompt families have fixtures. No reachable human choice is silently selected, and every failure path leaves a useful trace.

71–2 wks

Package the release candidate

Medium

Verify that the static production artifact, not only the development server, satisfies the MVP contract.

Outputs

  • One command for typecheck, lint, formatting, unit, integration, asset and browser gates.
  • Production-bundle browser smoke test: initialize, render an image, take an action and reach a result.
  • Chrome, Firefox and Safari desktop manual matrix.
  • Licensing/source-offer checklist and image-provider deployment decision.
  • Known limitations, rollback procedure and reproducible build instructions.

Gate 7 · MVP accepted

A clean checkout builds deployable static assets and completes the same deterministic duel without filesystem assumptions. Legal and image-host release blockers are either closed or deployment remains private.

05 · Delivery model

Organize around proofs, not component completion

The implementation should stay vertically testable. Parser, state, AI and UI work all follow a deterministic duel scenario rather than growing as disconnected subsystems.

Engine and Worker1–3Asset preload integrationWorker exclusively owns WASM and raw protocol.
Protocol and state3–6AI fixtures, minimal UIEvery protocol addition includes a fixture and typed public shape.
Opponent policy4, 6Final deck tuningOnly legal prompt options and legitimately visible information.
Svelte and Phaser4–5Image pipeline testsPresentation never becomes rules state.
Assets and updatesAlready active, then 1, 5–7All phasesOne atomic manifest, hashes, staging verification and rollback.
Quality and diagnosticsEvery phaseContinuousNo milestone closes without an automated or recorded proof.
Licensing and deployment1 and 7Provider validation during 2–6Public deployment cannot precede approval.

Planning estimate

The range is a solo-developer planning estimate, not a delivery commitment. It assumes the existing asset pipeline is retained, the synchronous WASM package is salvageable and the selected decks remain intentionally simple.

≈

18–30 engineer-weeks. The confidence interval should be recalculated after Gate 2. A core adapter rewrite, missing protocol support or complex deck choice can push beyond this range. Narrowing compatibility to preset-reachable messages is the strongest schedule lever.

06 · Risks and controls

Burn down unknowns in impact order

HighWASM adapter maturity

Own and pin the adapter. Gate the project at the deterministic headless Worker duel before complete protocol or UI investment.

HighProtocol breadth

Generate an inventory from pinned sources, classify every type and require a fixture whenever support is added or changed.

HighCore/data/script drift

Treat all revisions as one release unit. Reject partial snapshots before duel initialization and retain a previous known-good snapshot.

HighState divergence or secret leakage

Core remains authoritative. Reconcile with queries, assert invariants and project a separate public state before crossing the Worker boundary.

HighLicensing and card images

Keep image provider configurable. Separate technical availability from permission. Block public release without a documented posture.

MediumOpponent stalls

Choose simple decks, enumerate reachable prompts, include mandatory fallbacks and keep a hot-seat mode for diagnosis.

MediumStorage and host failures

Preload active-duel assets, use snapshot-aware caches and provide deterministic placeholders plus diagnostics.

MediumUI outruns engine truth

Make animations cancellable and presentation-only. Prompt IDs, not animation completion, control accepted responses.

07 · Definition of done

The MVP is accepted only when all outcomes hold

  1. Production browser assets initialize pinned ocgcore.sync.wasm inside a dedicated Worker.
  2. A verified atomic snapshot supplies all current standard-format card records, text, official scripts, required globals, strings and image references.
  3. The human and deterministic opponent can complete legal preset-deck duels from initial draw to a core-provided result.
  4. Typed prompt and response handling covers the accepted compatibility inventory, with fixtures and fail-fast diagnostics for unsupported input.
  5. Public state displays LP, turn, phase, chain and required zones without revealing hidden opponent information.
  6. Required card images appear in hand, field, inspector, GY, banished, Extra Deck and selection surfaces, with backs or reported fallback where appropriate.
  7. Surrender, restart and every tested finish path dispose the previous duel cleanly.
  8. A fixed seed and response sequence reproduce the ordered trace and final result.
  9. Asset integrity, typecheck, lint, unit, Worker integration and production browser smoke gates pass from a clean checkout.
  10. Errors download a structured trace containing revisions, seed, messages, responses, pending prompt and last successful event.
  11. Current desktop Chrome, Firefox and Safari pass the release verification matrix.
  12. Public deployment occurs only after AGPL/source obligations and database, strings and image-provider posture are documented.
