Cold startup

Fallback order during Worker asset load

  1. Fetch current packaged runtime with expected manifest SHA.
  2. If network/static load fails, open current snapshot from verified Cache Storage.
  3. If current cache fails, read active/previous snapshot pointers from IndexedDB.
  4. Attempt each fallback runtime cache using stored runtime manifest + image digests.
  5. If all fail, emit sanitized initialization error. Client watchdog also bounds no-response case.

Duel processing loop

Boundary-driven loop

DuelSession.processUntilBoundary() repeatedly calls synchronous core until WAITING or END. Controller then consumes complete message batch. It returns only at human prompt, result, or deterministic failure.

Hard limits

  • 10,000 core process iterations/session boundary.
  • 1,000 automatic opponent responses before human.
  • 128 pending Worker cmds.
  • 30s main-thread cmd watchdog.
  • 120s initialization watchdog.
  • 1s graceful disposal timeout.
  1. Core emits process status + parsed raw messages.
  2. Projector applies each msg → public state mutations + presentation events/result.
  3. PromptRegistry examines msg → domain prompt or null.
  4. Human prompt → stop loop, emit current snapshot/events/prompt.
  5. Opponent prompt → policy chooses opaque IDs from legal choices; registry encodes response; core resumes.
  6. Result → destroy core handle, preserve bounded trace, emit state/events/result.
  7. Waiting without supported prompt or end without result → fail as unsupported.

Opponent prompt

Opponent runs inside same controller loop. No Worker round trip. Policy is deliberately shallow, deterministic, deck-specific.

Input

Typed legal PlayerPrompt plus OpponentVisibleDuelState. Visible state contains counts/phase/LP, no card identities.

Decision

Priority: summon → special summon → activate → set → advance. Attacks choose strongest legal attacker. Optional chains/effects decline.

Evidence

Trace records prompt ID, opaque choices, player 1, machine-readable reason such as attack_strongest.

  1. PromptRegistry publishes player 1 prompt from core msg.
  2. Projector snapshot converts to identity-free opponent-visible summary.
  3. BasicOpponentPolicy picks legal choice IDs; sum/counter/sort prompts use specialized valid selection.
  4. Registry validates/encodes IDs to raw response; session calls core synchronously.
  5. Controller continues automatically until human boundary or result.

Snapshot/image activation

Runtime Worker and main-thread image loader operate in parallel. Local activation waits until both prove matching receipts.

  1. App stages composite activation ID with required receipts: runtime package SHA + active image manifest SHA.
  2. Worker loads runtime and emits runtime snapshot ID + selected image digest in ready.
  3. Main thread verifies active image manifest bytes, each image SHA/length/JPEG dimensions/decode, then builds object URL library.
  4. App waits for staged record, verified images, initialized core, expected runtime ID.
  5. Store verifies complete receipt set.
  6. Web Lock serializes cross-tab activation; IndexedDB CAS checks expected active ID/generation.
  7. Candidate becomes active; old active becomes previous; stale prior previous removed.
  8. Cache cleanup retains active/fallback runtime and image namespaces only.

If current runtime falls back, App requests image library for fallback snapshot/digest. Mismatched runtime/image identity never renders face images: deterministic placeholders/card backs preserve privacy and usability.

Diagnostics flow

Sensitive artifact. Download contains production seed. UI permits diagnostics only after inactive session, labels sensitivity, warns before sharing.
  1. Bounded Worker trace records lifecycle, process status, core msg type, presentation event, prompt, response IDs, opponent reason, result/error.
  2. After result/failure, user requests diagnostics. Client enforces initialized + prior session + inactive + single-flight.
  3. Runtime combines trace with core version and pinned revision metadata; computes pending prompt/last msg.
  4. Worker emits validated diagnostics event.
  5. App combines trace with build ID, browser info, active/fallback snapshot IDs, image cache summary.
  6. Downloader serializes schema v2 JSON to Blob, triggers local file download.
  7. App stores bounded metadata only (max 20 debug runs) in IndexedDB, not raw sensitive trace.
