Visual Novel Experience Prototype — TDD Implementation Plan

Status: draft for review
Scope authority: 09-experience-prototype-scope.md
Product decision: illustrated map is primary chapter-navigation surface
Delivery rule: each numbered section is independently testable and committable

1. Goal

Implement isolated, disposable visual-novel UX prototype without changing current direct-duel behavior or creating production campaign architecture.

Target flow:

Launcher
→ Title
→ New Game / mock Load
→ Narrative
→ Choice
→ Illustrated map
→ Pre-battle briefing
→ Mock battle outcome
→ Outcome scene
→ Reward/progression
→ Updated map
→ Save feedback
→ End/review

2. Mandatory TDD loop

Apply this loop to every committable section:

  • Write smallest automated test proving next behavior.
  • Run focused test; confirm failure for expected missing behavior.
  • Record RED evidence in commit notes or working log.
  • Implement minimum code needed to pass.
  • Run focused test; confirm GREEN.
  • Refactor only code introduced by current section.
  • Re-run focused test after refactor.
  • Run formatting, type, lint, unit, component, integration, build, reproducibility, browser gates through npm run check.
  • Confirm current direct-duel E2E remains green.
  • Commit section only after all gates pass.

Never write production behavior before failing test. Visual-only details must still receive semantic/component, state, responsive, or browser assertions before implementation.

3. Commit boundaries

  • Keep each numbered section in one standalone commit.
  • Do not mix unrelated cleanup, architecture migration, or duel changes into prototype commits.
  • Keep current index.html direct-duel entry behavior unchanged.
  • Keep prototype code under src/prototype/ except required root/config/test/doc files.
  • Do not import from src/duel/, src/worker/, src/field/, or src/storage/.
  • Do not add runtime deps unless implementation cannot proceed with existing Svelte/Vite/Vitest/Playwright stack.
  • Keep private-distribution safeguards intact.
  • Use suggested commit message or repo-equivalent conventional style.

Section 0 — Baseline evidence

Commit: none unless baseline documentation changes
Purpose: prove starting tree is green before prototype work.

Steps

  • Record current branch and working-tree status without changing unrelated files.
  • Record runtime snapshot ID and current test counts from latest trusted handoff.
  • Run npm run format:check.
  • Run npm run lint.
  • Run npm run typecheck.
  • Run npm test.
  • Run npm run build.
  • Run npm run build:reproducible.
  • Run npm run test:e2e.
  • Run npm run check as canonical baseline proof.
  • Stop prototype work if failure represents current baseline rather than planned RED test.
  • Save command/result summary in implementation notes.

Validation gate

  • Current direct-duel app initializes from root URL.
  • Existing Worker/privacy/browser tests pass.
  • No baseline repair is bundled into prototype commits.
