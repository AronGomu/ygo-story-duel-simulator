System in one diagram

App is static browser software. Main thread coordinates UI, local storage, image preparation. Dedicated Worker owns duel truth. Worker loads verified runtime closure, initializes synchronous WASM, translates raw engine protocol into clone-safe domain contracts. Phaser renders projections only.

app orchestration/boundary engine authority presentation/domain support

Non-negotiable architecture rules

Authority

Engine decides duel truth

ocgcore alone determines legality, effects, responses, results. UI, store, Phaser can validate UX constraints but cannot invent legal actions.

Isolation

Raw engine stays in Worker

Main thread never imports engine. Handles, raw protocol indexes, Lua/script maps, production seed remain Worker-owned.

Contracts

Only clone-safe domain data crosses

Commands/events use strict discriminated unions. Both sides parse, exact-key validate, bound payloads. Prompt/choice IDs are opaque.

Privacy

Hidden identities are absent

Opponent hand exposes count, not card objects. Face-down identity redaction occurs before main-thread boundary, not only in rendering.

Assets

Runtime data activates atomically

Engine, catalog, scripts, strings, images are one verified revision set. Runtime + image receipts must match before local activation.

Reliability

Failure is bounded, replaceable

Initialization, command, disposal, process loops have guards. Uncertain cleanup causes Worker termination/replacement, not reuse.

Where execution starts

  1. Browser mount. src/main.ts mounts App.svelte.
  2. Parallel preparation. App initializes Worker, opens IndexedDB snapshot store, preloads/validates active images.
  3. Worker initialization. Browser Worker loads manifest, active runtime closure, fallback cache if needed, then initializes vendored synchronous core.
  4. Automatic duel start. ready reduces store to idle; reactive app starts fixed MVP preset once per Worker generation.
  5. Engine loop. Worker creates production seed/session, processes core messages, projects state, auto-answers opponent prompts, pauses on human prompt or result.
  6. Dual presentation. Svelte renders authoritative semantic state/control surface; lazy Phaser bridge mirrors immutable snapshots/events.

High-attention areas

Large boundary files

App.svelte 1,286 lines; PromptRegistry.ts 1,245; Worker event parser 1,077. Changes can affect many states. Require focused tests.

Generated prerequisites

Vite config reads generated runtime manifest at config load. Fresh checkout needs npm run assets:mvp before dev/build.

Private deployment

Image redistribution remains unapproved. Normal production packaging refuses artwork. Approved command uses explicit private mode and writes marker.

Two runtime hosts

Browser Worker and Node Worker use shared runtime with separate asset loaders/bridges. Keep contracts host-neutral; test both boundaries.

Fallback complexity

Runtime + image fallbacks depend on synchronized snapshot metadata, digest receipts, cache namespaces, active/previous pointers.

Docs drift

Architecture docs are strong but status text differs. Verify statements against current tests/config before relying on historical plan docs.

Public protocol intentionally small. High-cardinality game protocol never crosses boundary directly.

Events: Worker → main

type DuelWorkerEvent =
  | Ready | Loading | State
  | PresentationEvent | Prompt
  | Result | Diagnostics | Error
  | Disposed;

Every event parsed by parseDuelWorkerEvent before store. Parser enforces clone bounds, exact keys, dense arrays, prompt player, public-state privacy.

Source: src/duel/contracts/duel-worker-event.ts

Boundary defense layers

  1. Sender validation. Client passes every command through parseDuelCommand before postMessage.
  2. Receiver validation. Worker parses untrusted message again before queueing.
  3. Session validation. Runtime checks initialized/active state and preset ID.
  4. Prompt validation. Only current prompt ID resolves; selection count/order/sum/place rules checked.
  5. Engine validation. Encoded response goes through vendored core.
  6. Outbound validation. Client parses every event; malformed/privacy-breaking event terminates Worker.
  7. Context validation. Store rejects events from old Worker/session generations.

State machines

Worker lifecycle

spawned → initialize queued → resources ready → one duel active → result/dispose → handle destroyed. Runtime can retain last bounded trace after session ends. Uncertain cleanup sets replacementRequired.

Snapshot lifecycle

staged → receipt-verified → active → previous → removed. Activation uses expected active ID + generation in one IndexedDB transaction. One previous snapshot retained for rollback.

Privacy path

Critical invariant. Opponent hidden card identity must never cross Worker boundary. Rendering a card back is insufficient if code/name remains in JS object.
  1. Raw core message may contain card code and field location inside Worker.
  2. DuelStateProjector applies visibility policy. Opponent hand gets handCount, but public hand array is empty.
  3. When visible card becomes hidden, projector rotates public instance ID and deletes code.
  4. PromptRegistry redacts prompt card code/name/description when viewer cannot see identity.
  5. Outbound event parser rechecks public state: opponent hand identity array must remain empty.
  6. Svelte/Phaser receive hidden cards without identity; Phaser selects card-back texture from hidden.
  7. E2E captures real Worker events and asserts opponent hand objects remain empty.

Cold startup

Fallback order during Worker asset load

  1. Fetch current packaged runtime with expected manifest SHA.
  2. If network/static load fails, open current snapshot from verified Cache Storage.
  3. If current cache fails, read active/previous snapshot pointers from IndexedDB.
  4. Attempt each fallback runtime cache using stored runtime manifest + image digests.
  5. If all fail, emit sanitized initialization error. Client watchdog also bounds no-response case.

Duel processing loop

Boundary-driven loop

DuelSession.processUntilBoundary() repeatedly calls synchronous core until WAITING or END. Controller then consumes complete message batch. It returns only at human prompt, result, or deterministic failure.

Hard limits

  • 10,000 core process iterations/session boundary.
  • 1,000 automatic opponent responses before human.
  • 128 pending Worker cmds.
  • 30s main-thread cmd watchdog.
  • 120s initialization watchdog.
  • 1s graceful disposal timeout.
  1. Core emits process status + parsed raw messages.
  2. Projector applies each msg → public state mutations + presentation events/result.
  3. PromptRegistry examines msg → domain prompt or null.
  4. Human prompt → stop loop, emit current snapshot/events/prompt.
  5. Opponent prompt → policy chooses opaque IDs from legal choices; registry encodes response; core resumes.
  6. Result → destroy core handle, preserve bounded trace, emit state/events/result.
  7. Waiting without supported prompt or end without result → fail as unsupported.

Human prompt response

Double-submit defense. Store marks responsePending; client records responded prompt IDs and clears current prompt before posting. E2E double-clicks same button, asserts one respond cmd.

Opponent prompt

Opponent runs inside same controller loop. No Worker round trip. Policy is deliberately shallow, deterministic, deck-specific.

Input

Typed legal PlayerPrompt plus OpponentVisibleDuelState. Visible state contains counts/phase/LP, no card identities.

Decision

Priority: summon → special summon → activate → set → advance. Attacks choose strongest legal attacker. Optional chains/effects decline.

Evidence

Trace records prompt ID, opaque choices, player 1, machine-readable reason such as attack_strongest.

  1. PromptRegistry publishes player 1 prompt from core msg.
  2. Projector snapshot converts to identity-free opponent-visible summary.
  3. BasicOpponentPolicy picks legal choice IDs; sum/counter/sort prompts use specialized valid selection.
  4. Registry validates/encodes IDs to raw response; session calls core synchronously.
  5. Controller continues automatically until human boundary or result.

Snapshot/image activation

Runtime Worker and main-thread image loader operate in parallel. Local activation waits until both prove matching receipts.

  1. App stages composite activation ID with required receipts: runtime package SHA + active image manifest SHA.
  2. Worker loads runtime and emits runtime snapshot ID + selected image digest in ready.
  3. Main thread verifies active image manifest bytes, each image SHA/length/JPEG dimensions/decode, then builds object URL library.
  4. App waits for staged record, verified images, initialized core, expected runtime ID.
  5. Store verifies complete receipt set.
  6. Web Lock serializes cross-tab activation; IndexedDB CAS checks expected active ID/generation.
  7. Candidate becomes active; old active becomes previous; stale prior previous removed.
  8. Cache cleanup retains active/fallback runtime and image namespaces only.

If current runtime falls back, App requests image library for fallback snapshot/digest. Mismatched runtime/image identity never renders face images: deterministic placeholders/card backs preserve privacy and usability.

Diagnostics flow

Sensitive artifact. Download contains production seed. UI permits diagnostics only after inactive session, labels sensitivity, warns before sharing.
  1. Bounded Worker trace records lifecycle, process status, core msg type, presentation event, prompt, response IDs, opponent reason, result/error.
  2. After result/failure, user requests diagnostics. Client enforces initialized + prior session + inactive + single-flight.
  3. Runtime combines trace with core version and pinned revision metadata; computes pending prompt/last msg.
  4. Worker emits validated diagnostics event.
  5. App combines trace with build ID, browser info, active/fallback snapshot IDs, image cache summary.
  6. Downloader serializes schema v2 JSON to Blob, triggers local file download.
  7. App stores bounded metadata only (max 20 debug runs) in IndexedDB, not raw sensitive trace.

Asset acquisition and transformation

Unified command: npm run assets:mvp. Offline regeneration: npm run assets:mvp -- --offline. Run is cross-process locked; final status becomes ready only after all verifiers pass.

Atomic snapshot identity

Runtime snapshot ID

Content-derived from verified asset manifest content digest + vendored engine manifest digest. Identifies rules/data/script compatibility unit.

runtimeSnapshotId = sha256(
  runtimeAssetContentSha256,
  engineVendorManifestSha256
)

Activation snapshot ID

Separates browser activation when image subset changes without runtime change.

activationId = sha256(JSON.stringify({
  runtimeSnapshotId,
  activeImageManifestSha256
}))

Why two IDs

Engine/data/scripts define duel compatibility. Active image manifest defines UI artifact set and legal distribution packaging. Combining them for browser activation prevents image-only release from colliding with existing persisted runtime record.

Browser data and persistence

Activation transaction

  1. Validate exact input shape, digests, runtime/image receipt consistency.
  2. Stage immutable candidate unless identical record exists.
  3. Verify receipts exactly equal required set.
  4. Open read-write transaction over snapshots + pointers.
  5. Validate active/previous pointer targets and generations.
  6. Compare actual active ID/generation to caller expectation.
  7. Move active→previous, candidate→active, update pointers to next generation.
  8. Delete stale previous after atomic commit.
Cross-tab safety. App combines Web Lock around activation/cache cleanup with IndexedDB compare-and-swap. Lock serializes cooperating tabs; CAS rejects stale actors even if lock unavailable/logic changes.

Production package flow

  1. vendor:verify proves vendored engine package/file hashes.
  2. snapshot:verify proves current runtime root and all generated inputs.
  3. vite build --mode private compiles Svelte, dedicated module Worker, sync engine adapter, lazy Phaser chunk.
  4. Vite runtime plugin copies only recursively resolved active-deck closure, active preset images, engine WASM/manifests, licenses.
  5. Unapproved image posture writes PRIVATE_DEPLOYMENT_ONLY.txt; ordinary non-private packaging refuses.
  6. build:verify independently rehashes source vs dist, exact file sets, content-derived IDs, engine chunk, Worker digest embedding, activation ID, licenses, forbidden Node markers, size budgets.
  7. build:reproducible creates two isolated builds and requires identical hashes.

Security and licensing boundaries

Security

Trusted local inputs only

  • Pinned project-provided Lua, decks, manifests.
  • Path containment + realpath checks in dev/build asset serving.
  • Strict manifest/payload bounds.
  • Dedicated Worker termination boundary.
  • Hidden identities removed before IPC.
  • No user scripts/decks or multiplayer authority in MVP.
Distribution

Public deployment blocked

  • Embedded core + CardScripts carry AGPL obligations.
  • BabelCDB redistribution terms need separate review.
  • Artwork/content/trademarks need permission.
  • Image provider access does not grant redistribution.
  • Current active manifest explicitly says redistributionApproved: false.
  • Build marker says keep artifact private.
Do not publish dist/ publicly. Existing build is intentionally private-only until source availability, DB terms, artwork/content permissions receive authorized review.

Quick launchers

Windows: download-mvp-assets.cmd. macOS/Linux: ./download-mvp-assets.sh. Both set project cwd, accept same flags as unified command.

Useful asset flags
npm run assets:mvp -- --offline
npm run assets:mvp -- --concurrency 8 --requests-per-second 12
npm run assets:mvp -- --force-images

--offline requires populated caches/archive. Default image concurrency/rate: 18; hard max rate: 20 req/s.

Verify environment incrementally

  1. Identity: node --version → 24+; npm --version; git --version.
  2. Generated status: generated/mvp-assets-status.json must say ready.
  3. Vendor: npm run vendor:verify.
  4. Data: npm run assets:verify + npm run snapshot:verify.
  5. Static quality: npm run format:check && npm run lint && npm run typecheck.
  6. Fast behavior: npm run test:unit.
  7. Real core: npm run test:integration.
  8. Browser package: npm run build, then npm run test:e2e.
Vite prerequisite. vite.config.ts reads generated/runtime/current/manifest.json immediately. Missing generated assets can fail config loading before dev server starts.

Mental model in 90 seconds

Rules

Core owns truth. UI never calculates legality. Worker protocol maps engine prompts to opaque choices.

Isolation

Worker owns secrets/handles. Main thread sees immutable redacted state and current human prompt.

Presentation

Svelte is complete UI. Phaser adds spatial visuals; failure cannot block duel controls.

Data

One compatibility snapshot. Engine/catalog/scripts/strings/images are pinned, hashed, activated together.

Reliability

Replace, do not trust uncertain state. Hung/broken Worker gets terminated and recreated.

Evidence

Headless first. Real-WASM fixtures/transcripts prove game support before UI polish.

Safe change recipes

Add a supported engine prompt family
  1. Add/verify engine message classification and stable fixture.
  2. Add domain PromptKind/choice shape only if existing model cannot express it.
  3. Map raw msg to opaque choices in PromptRegistry.
  4. Encode/validate response, retaining raw indexes only in registry binding.
  5. Add client-side prompt selection rules if multi-select/sum/order semantics differ.
  6. Add accessible control family in PromptControls.svelte.
  7. Add policy branch so opponent can always answer reachable prompt.
  8. Update exhaustive switches and event/contract parsers.
  9. Write unit fixture, component test, programmed real-WASM scenario, keyboard E2E only if browser path newly distinct.
Add field presentation for existing event
  1. Keep event domain-level; no raw core values.
  2. Map event to PresentationCommand with reduced-motion duration.
  3. Render cancellable effect in DuelScene.present.
  4. Ensure applying snapshot remains idempotent; event is cosmetic.
  5. Ensure restart/dispose cancels timers/tweens and releases refs.
  6. Test command mapping and bridge disposal/failure.
Change public duel state
  1. Define privacy requirement first.
  2. Change source contract and exact-key runtime parser together.
  3. Update projector and invariants.
  4. Update event parser privacy checks.
  5. Update store/UI/field consumers.
  6. Add structured-clone/round-trip tests, projection fixture, hidden-information test.
  7. Run Node Worker integration and browser event-capture assertions.
Update pinned assets or engine
  1. Use isolated branch; update refs/version/integrity intentionally.
  2. Regenerate full asset snapshot and runtime snapshot.
  3. Review count/protocol/active closure drift; never accept blind trace update.
  4. Run all protocol classification + programmed real-WASM suites.
  5. Run private build, independent verifier, reproducibility check, browser matrix.
  6. Archive exact manifests with release candidate; keep previous snapshot available.
  7. Repeat legal/distribution review when source/image posture changes.
Fix bug
  1. Classify layer: UI, projection, protocol, core/data, asset, storage, lifecycle.
  2. Capture fastest evidence: Worker event, downloaded trace, failing fixture, cache/manifest digest.
  3. Write smallest deterministic test reproducing failure before fix.
  4. Fix owning layer only. Do not hide projection/protocol bug in Phaser/UI.
  5. Run direct suite, then boundary integration, then broader gate based on affected contracts.

Worker log vocabulary

  • duel.worker.command.* — receive/reject/fail/complete/skip.
  • duel.worker.event.* — event dispatch/post failure.
  • duel.worker.detached — attachment teardown.
  • duel.worker.node.* — Node Worker bootstrap/thread boundary.
  • duel.worker.logging.failed — logger fallback failure.
  • duel.client.worker.* — main-thread creation, boundary failure, disposal/replacement.
  • duel.field.* — lazy Phaser startup/texture failures.
  • duel.app.* — storage activation/cleanup and destroy failures.

Diagnostic file

Request after duel result/failure. File contains app/browser/build metadata, active/fallback snapshot IDs, image cache counts, core version, upstream revisions, production seed, ordered trace. Treat as sensitive. Reproduce with exact snapshot + seed + ordered responses in authorized local workflow.

Common traps

Import engine on main thread

Breaks isolation and packaging. Only OcgCoreAdapter.ts imports vendored package.

Use Phaser as state truth

Sprites are disposable projections. Query public state/store instead.

Hide instead of redact

Card back does not protect code in object. Remove identity before Worker boundary.

Fetch inside core callback

Callbacks are synchronous. Preload all card/script data into Worker maps.

Expose raw indexes

UI uses namespaced opaque prompt/choice IDs. Raw response indexes remain private binding data.

Reuse uncertain Worker

Cleanup failure means terminate + replace. Never continue on possibly live handle.

Run Vite before assets

Config needs generated manifest. Complete asset baseline first.

Update traces to make CI green

Understand semantic drift. Fixtures are compatibility evidence.

Package full data archive

Production includes recursively resolved active closure only.

Publish private dist

Current images/content lack approved redistribution posture.
